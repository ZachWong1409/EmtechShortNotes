Topic 1 — Intro to Data Science

foundations
Theory (Q1)
Data science = interdisciplinary field extracting knowledge from structured/unstructured data
"4th Paradigm": Observational → Theoretical → Computational → Data-intensive discovery
  • Big Data = "crude oil" (raw); Data Science = "refinery" (insight)
  • "Data science produces insights. ML produces predictions."
  • Holistic approach: Data Mgmt, Modeling, Visualization, Trustability — wrapped by Ethics/Privacy
Application (Q2)
  • Fraud detection — early detection harder; false pos/neg both costly
  • Recommender systems — Netflix $1B/yr, Amazon +20-35% sales lift
  • Patient readmission — ties to socioeconomic + genetic data
  • Smart cities — planning, asset mgmt, open data
  • Moneyball — $40M budget beat Yankees' $120M via sabermetrics

Topic 3 — Data Mining Applications & Systems

domains + system choice
Theory (Q1)

Choosing a DM system — 5 criteria:

  1. Data types supported (relational, text, transactional, spatial)
  2. Coupling: no → loose → semitight → tight (ideal)
  3. Scalability: row vs column ("curse of dimensionality" — column is harder)
  4. Visualization tools
  5. Range of functions & methods offered

Data mining sits at the "Chasm" in tech adoption — systems too generic, need business-specific integration

Application (Q2)
  • Biomedical/DNA — association analysis: diseases triggered by gene combinations, not single genes
  • Financial — money laundering detection via multi-DB integration (transactions + crime history)
  • Retail — sequential pattern mining on loyalty card data to spot declining customer loyalty
  • Telecom — multidimensional fraud pattern detection (call time, duration, location)

Topic 4 — Data Warehouse

OLAP / schemas
Theory (Q1)

4 properties (Inmon's definition): Subject-oriented · Integrated · Time-variant · Non-volatile

OLTP = run the business now (current, normalized, fast read/write)
OLAP = analyze the business over time (historical, denormalized, complex queries)

3 schemas: Star (1 fact + radial dims) · Snowflake (dims normalized further) · Fact Constellation (multiple fact tables share dims)

Application (Q2)
  • Roll-up (drill-up) — summarize, climb hierarchy (day→month→year)
  • Drill-down — reverse, more detail
  • Given a sales-by-region/time scenario → pick schema based on dimension complexity (snowflake if dims need normalizing, star if simple)
  • Enterprise warehouse (whole org) vs Data Mart (one dept/group)

Topic 5 — Data Analytics Lifecycle

process / roles
Theory (Q1)

6 phases, in order:

  1. Discovery — learn domain, assess resources, frame problem
  2. Data Prep — build sandbox, ETL/ELT, condition data
  3. Model Planning — explore variables, pick technique
  4. Model Building — train/test/production datasets
  5. Communicate Results — success/fail vs Discovery criteria
  6. Operationalize — pilot, deploy, deliver reports/code

CRISP-DM (6 phases): Business Understanding → Data Understanding → Data Prep → Modeling → Evaluation → Deployment

KDD is broader than data mining — DM is one step inside KDD

Application (Q2)

Roles: Business User, Project Sponsor, Project Manager, BI Analyst, Data Engineer, DBA, Data Scientist

4 Core Deliverables: Sponsor presentation (big picture/ROI) · Analyst presentation (technical/ROC curves) · Code · Technical specs

Case-study answer pattern: well-maintained data + experienced team → less time in Discovery/Data Prep, more in Model Planning/Building → justify lifecycle or CRISP-DM choice based on data structure & team maturity

Topic 6 — Data Preprocessing

cleaning / transformation
Theory (Q1)

6 quality dimensions: Accuracy, Completeness, Consistency, Timeliness, Believability, Interpretability

4 major tasks: Cleaning · Integration · Reduction · Transformation

Missing data fixes (simple→smart): ignore tuple → manual fill → global constant → attribute mean → class-mean → most probable value

Noisy data fixes: Binning · Clustering · Regression · Human+computer inspection

Application (Q2)
  • Schema integration issue: entity identification problem — e.g. cust-id vs cust-#, same entity different names
  • Scenario: missing income data, segmented by customer class → use class-mean over plain mean (respects segment differences)
  • Transformation types: Smoothing, Aggregation, Generalization (concept hierarchy), Normalization
